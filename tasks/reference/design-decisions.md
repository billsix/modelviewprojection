# modelviewprojection design decisions (digest)

**Reference document** — a distilled index of the durable design decisions and rationale, pulled from the archived task docs so they aren't lost in the don't-trawl archive. Not a task; update in place. Last updated 2026-07-21.

This complements `CLAUDE.md` (the current rules) by recording **why** each rule exists and **where** it was decided. Paths are repo-relative under `tasks/archive/`. Where a decision was parked, declined, or reverted, that is stated — those are the ones most likely to be re-litigated. When two docs disagree, the later date and `CLAUDE.md` win; the earlier entry is kept for the reasoning.

## Math library & gacalc integration

- **mvp's home-grown vector/transform math was deleted; gacalc's `Vector1/2/3` + transform layer became *the* vector type.** "Switch the type, not just import helpers" (Option B) to remove maximal duplication; a compat `Vector3D` subclass was rejected as reintroducing a second vector type. Gotchas that bite every consumer: fields are `coeff_e_1/2/3` (not `.x/.y/.z`); `dot()` returns a *multivector* → `.scalar_part()`; `cross` is `(v^w).dual()` (check sign vs CCW tests); `__iter__` yields per-blade multivectors, so vertex unpacking must use the graded types. Sourcing bare `e_1` from `gacalc.g2` yields a full `G2` (loses the graded type + fast closed-form sandwich) — `Vector2.e_1` is mandatory. Book Phase 4 (teach gacalc vectors in ch05/06/14) relocated to `tasks/book-rotate-prose-update.md`. (`2026/07/09/gacalc-math-migration.md`)

- **`mathutils.py` is now only mvp-local graphics helpers; the pass-through façade re-exports were removed (2026-07-21).** Bill's rule: no re-export façade — anything referenceable in gacalc imports directly from gacalc. The whole function/transform layer imports from `gacalc.transforms` (even names defined in `gacalc.functions` but re-exported there); only value types split by module (`base`/`g1`/`g2`/`g3`). Bonus: matches the book's doc-region listings, which already show these from gacalc's source. Surfaced pre-existing ch10 rot (`Vector2.e_1()`→`.e_1`, `scale_non_uniform_2d`, `.isclose`). (`2026/07/21/remove-mathutils-gacalc-facade.md`; migration groundwork in `2026/07/09/gacalc-math-migration.md`)

- **`rotate`/`rotate_x/y/z` are direct bindings of gacalc's `plane_rotation(a, b)`, plane derived once at import — not per-call wrappers.** The old façades hand-built `cos θ·a + sin θ·b` and re-derived the plane every call; `cayleyscene` calls `.at(t)` per edge per frame. Deriving the plane once: **160.6µs → 6.4µs (25×)**, and returns plain `float` instead of leaking a sympy `Float` (which had made every downstream demo op ~6× slower). `rotate_90_degrees` retired (= `rotate(radians(90))`). Orientation predicates moved into their sole consumer, the software rasterizer, computing the wedge `(v1^v2).coefficient(Bivector2.e_12)` directly. (`2026/07/09/plane-rotation-mathutils-adoption.md`)

- **`is_parallel` → `is_parallel_and_same_orientation` (behaviour unchanged).** The old name disagreed with the GA sense — the predicate also requires same orientation, so the honest name states both. Deliberately did **not** switch the guard to a wedge test (Option B, deferred by Bill); the precise name + a call-site comment make the latent degenerate/anti-parallel-triangle gap *visible* instead of silently fixed. (`2026/07/19/is-parallel-ga-semantics.md`)

- **`shading._face_normal` reimplemented on `mathutils.find_normal` but keeps its tuple return.** It duplicated `find_normal`'s cross-product/CCW formula bit-for-bit (max diff 2.2e-16). The tuple return is deliberate: the GL boundary and indexing callers (`light_dir[i]`) want tuples, and adding `__getitem__` to the vector type was rejected — the course intentionally avoids treating vectors as indexable arrays. (`2026/05/26/shading-use-vector3d.md`)

- **Scale-transform inverse LaTeX renders the reciprocal `\frac{1}{m}`, not `S_{-m}`.** Matches `f_inv = vector*(1/m)`, applied consistently to all three scale inverses. Gotcha for re-introduction: `\\\frac` inside an f-string is backslash + form-feed (`\f`) + `rac`, not `\frac`. Latent because nothing rendered/tested inverse LaTeX. (`2026/05/26/fix-mathutils-latex-and-error-strings.md`)

- **`is_clockwise` was infinite-recursing** (called itself instead of `not is_counter_clockwise(...)`); fixed + regression test. Hidden indefinitely because it had no test and its `doc-region` marker was unused in any chapter — never exercised. The lesson: predicates with no test and no book-slice can harbor stack-overflow bugs forever. (`2026/05/26/fix-is-clockwise-recursion.md`)

## The matrix / function stack

- **`pyMatrixStack.py` renamed to `matrix_stack.py`; both `N813` ruff exemptions deleted at the source.** The alias `import … as ms` was never the problem — the camelCase *module name* was what `N813` flagged. Done in **one commit, no compat shim**: the module holds mutable module-level state (`__modelStack__` etc.), so a double-import under two names would create two independent stacks. Zero book references. (`2026/07/19/rename-pymatrixstack-module.md`)

- **Five latent matrix-stack bugs fixed, incl. `get_current_matrix`'s silent `None` fall-through (five `if`s, no `else`) — now a `match` with `case _: raise`.** This is *the* worked example behind the "prefer `match` + `case _`" rule: every real `MatrixStack` member was handled, so it looked fine; the hole opened only for an unhandled member, and every caller indexed the result. Also: the `np.matrix`→`np.ndarray` migration is **not** find-and-replace — `multiply` does in-place slice assignment that behaves differently for `np.matrix` (which overloads `*` as matmul). Equivalence proven by *mechanically* reverting the one change (`np.array(`→`np.matrix(`) into a second module and diffing (max |Δ|=0.0), after a hand-written reference had earlier introduced a phantom 14.5-unit regression. Note the read/write asymmetry: reading a *derived* stack product still works; only *writing* one now raises. (`2026/07/18/pymatrixstack-bugs.md`)

- **The `mathutils.fn_stack` module global was silently splitting into two objects — root cause was a packaging omission, not the global.** With no top-level `src/modelviewprojection/__init__.py`, `find_packages(where="src")` returned `[]` and pytest imported modules under bare names, double-importing each run. The global is **deliberate pedagogy** (the Python analogue of OpenGL's implicit matrix stack, `literalinclude`d into ch16) — removing it / ContextVar / explicit-param were all rejected as perturbing taught code. Instructive dead-end: a loud import-guard (option E) was implemented and **reverted** — it couldn't distinguish pytest's harmless double-import from a harmful one (identical `sys.modules` signatures). The real gate found what an isolated prototype couldn't. Same `__init__.py` fix applied to gacalc. (`2026/07/19/fn-stack-global-fragility.md`)

## Book & code-sync

- **The book includes code by MARKER (`doc-region-begin/end` + `:lineno-match:`), never by line number.** Editing a source file cannot "break" line numbers — Sphinx recomputes them at build. Consequence for drift audits: `literalinclude` snippets are **always live** (build-synced), so drift only ever lives in the *prose, captions, and hand-written doctest blocks* around them — that's where to look. `translate` is a module-level function returning an `InvertibleFunction` (vector addition wrapped), **not** a `Vector.translate()` method — book prose had claimed otherwise. (`2026/05/27/book-code-drift-ch1-6.md`; the marker mechanics are now in `CLAUDE.md`)

- **Some book listings are `literalinclude`d from GACALC's own source, docs-only (2026-07-20).** The vector/transform math lives in external gacalc, not this repo; the book still shows it via gacalc's doc-region markers. Two PyPI artifacts, same version: the **wheel** is the runtime dep, the **sdist** is unpacked to `/opt/gacalc-src` docs-only (never on `sys.path`, chosen over a git clone because it already carries the generated `g1/g2/g3/scalar.py` with markers baked in). The doc-region checker moved **into the container** because `_gacalc_src` only exists in the image. Editing such a listing means editing gacalc and releasing it. (Now in `CLAUDE.md`; open follow-up `tasks/dangling-book-code-includes.md`)

- **The crossproduct demo was removed entirely** (`mathdemos/`, `notebooksrc/crossproduct.py`, `tests/test_crossproduct.py`) — it moved to the multivariate-math repo. Side effect: `ty check src` went **79 → 0**. texExpToPng build stays (the book's inlinetex extension uses the binary, independent of the demo). (`2026/07/09/remove-crossproduct-demo.md`)

## Ports — Code-the-Classics games

The 10 CtC ports sit on one clean-room `pgzero_gl` shim (GLFW + OpenGL 3.3 core). The **rule-set evolved**, so date order matters:

- **Original rule (2026-06 → 06-29): behaviour-faithful ports, annotations only, NO restructuring, NO ruff.** Games were still treated as line-by-line BSD-derivative copies diffable against upstream. Type-clean via `ty`; nullable-but-unguarded Actor attrs typed `Any` not `Optional` (games deref assuming non-null; `Optional` just relocates the error). (`2026/06/25/port-codetheclassics-vol1.md`, `.../vol2.md`, `2026/06/29/codetheclassics-types-and-docstrings.md`)

- **Rule relaxed (2026-07-08): behaviour-faithful, but STRUCTURE may modernize** — dataclasses, `match`, `@override`, precise types, ruff-formatted. Bill's call, because he owns all 10 games. "Behaviour-faithful" still means **same RNG call order, same update/draw order** (byte-identical gameplay). This is the doc that rewrote the `CLAUDE.md` rule. (`2026/07/08/ctc-dataclasses-and-dispatch.md`)
  - **Dataclass rules:** always `@dataclass(eq=False)` (generated `__eq__` would swap identity for field equality — games rely on identity). **Never dataclass a class whose base or subclasses are also dataclasses** (InitVars merge, breaking positional call sites) — convert stateful leaves over plain bases only. Keep explicit `__init__` when super-args are RNG-computed (preserves RNG order → byte-identical). The heterogeneous update/draw loops and ABC families are **deliberate curriculum** (the books teach OOP), not pgzero leftovers — keep them. (`2026/07/08/ctc-dataclasses-and-dispatch.md`)
  - **`match` only where EVERY branch is `subject == Enum.MEMBER`;** chains with a trailing non-enum branch stay `if/elif`. Enum cases must be **dotted** (bare name = capture-pattern footgun). Preserve deliberate upstream no-op fall-throughs. `@override` is high-value here — the whole dispatch hangs on override links, so it turns a renamed/typo'd base method from silently-dead dispatch into a type error (98 applied). (`2026/07/08/ctc-match-and-modern-python.md`, round 2 `2026/07/09/ctc-modernization-round2.md`)
  - **Enums that do arithmetic must be `IntEnum`, not `Enum`** (myriapod indexes/does modular math on `Direction`; `str(IntEnum)` since 3.11 is the plain number, so sprite-name construction stays byte-identical). Deliberately left as plain ints: soccer's 0-7 compass `dir` (arithmetic-native — an enum would fight the mechanic), kinetix brick-type ints (level data, not symbolic states). (`2026/07/09/ctc-modernization-round2.md`)

- **"Honest imports" cluster (2026-07-08): make imports reflect reality now that Bill owns the games** (drops the byte-faithful-to-upstream constraint; behaviour-faithful retained). Deleted the synthetic `pygame`/`pgzero` `sys.modules` impersonation modules (~130 lines) and their `ty: ignore[unresolved-import]` suppressions; replaced `from pgzero_gl import *` with explicit subsets (unmasking caught a real latent `gldraw.` NameError in soccer); hoisted function-local imports except the deliberate GL/PIL/renderer defers. Trap: the shim re-exports pgzero's `exit()` — an explicit import must include it or the builtin silently wins. (`2026/07/08/ctc-honest-imports.md`, `ctc-explicit-star-imports.md`, `ctc-imports-at-top.md`)

- **The shim `Vector2` subclasses `gacalc.g2.Vector2`; `vec * vec` RAISES.** pygame's `*`-means-dot clashes with gacalc's geometric product, so every `vec*vec` site was first converted to explicit `.dot()` (Phase A) and `Vector2.__mul__` made to raise on a vector operand — a missed site fails loudly, never silently changes meaning. Subclassing required an upstream gacalc generator change (subclass-preserving `type(self)(...)` ops) *and* a gacalc relicense to LGPL. Two permanent documented `invalid-method-override` variances (`dot`, `rotate`). Benchmark gate: 2.6%→3.4% of a 60fps frame at 200 actors — subclass accepted over facade. (`2026/07/08/ctc-vector2-deferral.md`) — **Note:** `CLAUDE.md` (2026-07-09) now says the games use `gacalc.g2.Vector2` *directly* with no shim vector subclass; the subclass here was superseded the next day. (`2026/07/09/codetheclassics-gacalc-math.md` records the cautious shim-delegation plan that was *exceeded* — gacalc grew `x/y/z` properties + quotient `/` in 0.0.8 instead.)

- **The shim's `Actor.__getattr__`/`__setattr__` magic was DELETED** and replaced with real static properties. Upstream pgzero uses open-world attribute magic because it teaches children; the 10 games are a **closed, enumerable corpus** (play-instrumented: zero getattr-misses across every game), so the machinery was pure per-frame overhead (plain write 844→10ns, 84×). Kept dynamic only where genuinely open + not hot (`keyboard`/`images`/`sounds`, first-touch cached). Incident: the new `pos` property collided with subclass `pos: InitVar` fields → renamed the InitVar to `spawn_pos` everywhere; added a **definitions gate** (`runpy.run_path` with `go` stubbed) because `py_compile` doesn't execute class bodies. Caught ~20 latent bugs. (`2026/07/09/ctc-shim-dynamism-audit.md`; local-type extension `2026/07/09/ctc-more-types.md` — module-level `game` typed `cast("Game", None)` with a documented "None only on title screen" invariant, chosen over 120 narrowing guards after a Fedora `ty` bump got stricter)

- **Audio: one miniaudio `PlaybackDevice` + a software-mixing callback (pygame.mixer's architecture), NOT one OS stream per voice.** Dropped `just_playback` for `miniaudio`. Root cause (found by diffing against untouched upstream + library source): just_playback opens one OS audio stream per voice, kept open forever to dodge a no-finalizer leak; leadingedge crossfades 40 looping engine samples → dozens of concurrent ALSA/PipeWire streams → clunk (separate devices can't sample-level crossfade) + freeze (stream-slot exhaustion blocking the game thread inline). An earlier 50Hz fade-thread fix was **disproven as the cure** (both symptoms persisted) but kept as correct semantics and folded into the callback. Now exactly one OS stream ever, so the freeze mechanism is structurally impossible. Not `pygame.mixer` because host SDL2-on-SDL3 breaks SDL audio; headless → graceful no-op. (`2026/07/09/leadingedge-audio-clunk-and-freeze.md`)

- **A second fixed-function `renderer_gl1.py` backend, selectable via `PGZERO_GL=1`, pixel-identical to the 3.3 renderer.** Runs on old/limited GL (software GL, Pi, remote X); also pedagogical — mirrors the book's demo19 fixed-function era. Tractable because the renderer was already behind a small interface (`draw_image`/`fill`/`set_clip`/…). Offscreen `pygame.Surface` is emulated as a **CPU RGBA buffer, not an FBO** — the games only need fills + sprite accumulation. (`2026/06/25/pgzero-gl-opengl1-renderer.md`, `.../port-codetheclassics-vol2.md`)

## Ports — SuperBible v4

- **Ported per-frame `sin`/`cos` immediate-mode mesh generation into shared `_primitives.py` builders that precompute once — but rendering stays immediate-mode.** Explicit constraint: **NO display lists or VBOs unless the C++ source already uses them** (faithfulness over performance); only *when the trig runs* changes. The plan's "one `build_cone` with optional stacks" was wrong on contact (three distinct tessellations exist) — builder makes only the shared fan body, demo-specific caps stay local. `build_octahedron` dropped (hardcoded literals, no trig, 3 inconsistent variants — couldn't be byte-faithful). Verified byte-identical via a GL-stub call recorder. (`2026/05/29/extract-data-generation.md`)

- **The bulk walk-around-camera migration was attempted and REVERTED — a cautionary tale.** Two failures: (1) a regex used to delete function bodies (`(?:\s+[^\n]*\n)+`) ate `def main()` across 33 files because `\s` matches `\n` — "compiles clean" but `NameError` every frame; rewritten with `ast.parse` + `lineno`/`end_lineno`. (2) The camera itself was pedagogically wrong: **object-rotation demos (`x_rot`/`y_rot`) are orbital framing, not camera movement** — replacing "spin the jet" with "fly loops around the jet" is a worse control scheme; 2D `glOrtho` demos shouldn't get a 3D camera at all; no single `move_speed` fits all scene scales. Prescriptions: don't migrate object-rotation demos; hand-migrate one-per-pattern and verify visually before any bulk pass; grep-confirm a plan's "done" claims (the menubar plan claimed 92 migrated files; audit showed 0). (`2026/05/26/notes-2026-05-02-v4-camera-attempt-reverted.md`; postmortem `2026/04/28/postmortem-phase2-attempt-1.md`)

## Tooling, types & gates

- **`format.sh` fails on ANY step (every step always runs, script exits nonzero if any failed).** The old exit-code-of-last-command behaviour masked red steps — 79 `ty check src` diagnostics were visible in output but `make format` exited 0 because a passing check ran last. (`2026/07/09/src-ty-diagnostics-after-ty-bump.md`)

- **`ty check src`/`tests` driven to 0; `make shell` exits 0.** ~95% of the ~177-diagnostic flood was one root cause: gacalc 0.0.3 typed coefficients as `numbers.Real`, which stricter `ty` correctly rejects for plain `int`/`float` (the numeric tower is a runtime ABC, not a static subtype). Fixed by releasing gacalc 0.0.4 with `Coef = int | float | sympy.Expr` and bumping the pin — the container ty run also caught two real runtime bugs the bump would have caused. Because ty runs last in the exit-trap, driving it clean made an "advisory ty" mode unnecessary. (`2026/06/13/shell-exit-ty-check-noise.md`)

- **Doctests run as part of the suite (`--doctest-modules`), but only where they document a contract.** Before, `pytest.ini` lacked `--doctest-modules` and `testpaths` pointed only at `tests/`, so four doctests had failed silently since NumPy 2 changed scalar repr (`np.float64(-4.0)` vs `-4.0`) — fixed by printing plain floats. GUI/OpenGL/plotting code is deliberately **not** doctested (a doctest there is "theater" — opens a window, asserts nothing). Don't write doctests to inflate coverage. (`2026/07/19/doctests-everywhere.md`)

- **The main-guard + `:dedent:` plan (main-guard 25 demos, add `:dedent:` to 129 book includes) was CLOSED as not-needed.** Its premise was false: the `pytest.ini` allow-list it aimed to remove already names every library area, so doctests can cover 100% of the library today with zero restructuring. Paying 25 reshaped files + 129 edits to enable the least-valuable doctest location is a bad trade. (`2026/07/19/demo-main-guards-and-dedent.md`)

- **Moving the runnable demos out of the package into top-level `demos/`/`visualizations/` is PARKED — not approved, do not start.** It was drafted as a (false) prerequisite of the doctest work, which was already complete — classic goal drift ("deep in inception … a huge rewrite without keeping the goal in sight"). If ever revived it must be justified freshly on **student discoverability**, and its recorded "yes" answers are NOT approval. The measurements are kept because they were expensive: 40 files execute on import; 129 of 181 `literalinclude`s point at `demos/`; nothing imports `demos/`; the June `sys.path` fix survives the move *only* because the engine (`_pipeline.py`/`cayley_gl.py`/`cayley/`/shaders) stays in the package. Blocker: `[project.scripts]` entry points are incompatible (0 of 31 demos define `main()`; adding one destroys the flat top-to-bottom script the book displays) — recommendation is to drop entry points, since `python demos/demoNN.py` is what the README already teaches. (`tasks/move-demos-out-of-package.md` — ACTIVE, parked)

- **texExpToPng un-vendored — Dockerfile builds it from a SHA-pinned git clone** (`fbbd9a3f…`), killing the three-copy sync burden. Bump the SHA deliberately. Gotcha caught by Bill's first `make html`: `book/docs/_static/Makefile` had its own build-from-vendored-dir rule that had to be removed. (`2026/07/08/unvendor-texexptopng.md`)

- **PDF build uses `latex_engine = "lualatex"` (+ `latex_use_xindy = False`).** A `√` (U+221A) leaking from autodoc'd gacalc docstrings killed pdflatex on "Unicode character" — fixed permanently by the engine switch rather than escaping glyphs. The stale-`.aux` "Runaway argument / `\@newl@bel`" error is cured by `rm -rf _build/latex`. (`2026/07/08/finish-pdf-epub-build.md`)

- **Interactive demos run in-container over Wayland by pointing `PYGLFW_LIBRARY` at Fedora's *dual* X11+Wayland `libglfw.so.3`.** The naive `PYGLFW_LIBRARY_VARIANT=wayland` is WRONG and crashes: PyPI glfw registers a Wayland-only lib (no `glfwGetX11Window`), then imgui_bundle's native lib binds it and dies on the missing X11 symbol. The dual lib satisfies imgui_bundle's X11-symbol need while GLFW still picks Wayland at runtime. No X server/xauth needed. (`2026/06/15/run-demos-in-container-wayland.md`)

- **`image-export`/`image-import` standardized** (`$(CONTAINER_CMD)`, `.PHONY`, tar gitignored) — mvp was the *original* home of the pair but predated the convention; the multi-hundred-MB export tar was one stray `git add` from being committed. (`2026/06/13/fix-image-export-import-gaps.md`)

- **`tests/*.py` fully annotated (`-> None` on 63 tests, real helper signatures, ~177 locals).** Two gotchas: `assert_same_fn` params must be `Callable[..., MultiVectorBase]` not `Callable[[…], …]` (Callable args are contravariant — ty caught 8 errors); locals were typed via a static "MonkeyType" method (`reveal_type` → run ty → clean the inferred types back in). Bill's steer: readability, so locals typed after he asked (initially left inferred as churn-not-clarity). (`2026/07/20/type-annotate-tests.md`)

- **`*iter(x)` → `*x`** across 34 sites: `*` already iterates, so `*iter(x)` iterates twice — a *consistency* fix converging on the `*vector` form taught right beside these calls. (`2026/07/20/simplify-redundant-iter-unpacking.md`)

## Packaging & layout

- **The flat ~40-file package was regrouped into `demos/` / `util/` / `framebuffer/` subpackages — hard break of old import paths, no compat shims.** `mathutils.py` + the matrix stack stay at package top: moving a *utility* churns every importer (~130 sites), while moving a *demo* is cheap (only book `literalinclude` paths change, not its imports). `make html` is the hard guard — a wrong path fails the Sphinx build loudly. (`2026/06/03/restructure-directories.md`)

- **`mvpVisualization/` moved into the package as `mvpvisualization/`, `sys.path` hacks replaced with absolute imports.** Demos run **by file path, not `python -m`**, which *forces* absolute imports (relative imports fail in a path-run `__main__`). Deliberately no `package-data`/wheel work and no `main()` wrapper: shaders load `__file__`-relative and these modules are only run, never imported. This is the fix the parked demos-out-of-package move must not undo. (`2026/06/03/move-mvpvis-demos.md`)

## Render pipeline & GL conventions

- **The Cayley graph is an explicit immutable/acyclic data structure (`CayleyGraph`/`Edge`/`path`) + a headless scene/animation engine; all seven viz demos rebuilt on it.** Load-bearing seams: interpolation is an optional law on `InvertibleFunction` (composites recurse via `components`/`steps()`, not stored closures), and `inverse` *commutes* with `at(t)` so backward-walked edges animate. Camera placement == object placement (declare the camera edge once, walk it backward → one graph drives both imgui trees). The perspective/ortho squash stays a GPU shader step modelled as `NonInvertibleTransformation`, **not** an affine edge, because it's projective. `Frustum` (perspective) and `RectangularPrism` (ortho) are separate types — "the ortho is not a frustum." (`2026/06/03/cayley-graph-visualizations.md`)

- **A default VAO is bound at startup and never unbound; VBO/VAO construction split into `make_vbo` + `make_vao([AttribSpec])`.** Fixes real macOS Core Profile support — VAO 0 is prohibited there. The original bug diagnosis ("shaders won't compile") was WRONG: compile/link need no VAO; the real failures are `glVertexAttribPointer`/`glEnableVertexAttribArray`/`glDraw*`. `AttribSpec` keeps `(stride, offset)` a coupled tuple deliberately (splitting them is a foot-gun). Shared-VBO dedup shows up naturally as repeated `vbo=` in the spec list. (`2026/05/27/vao-vbo-separation.md`)

- **Loose per-pipeline `program`/`u_*` globals grouped into frozen `@dataclass` pipeline structs — plain data containers only, no methods.** The `u_bs_*` prefix convention was already doing a struct's job; the dataclass makes program↔uniform ownership explicit without an OOP upgrade (keeps free functions free / procedural curriculum style). Students already know `@dataclass` from `mathutils.InvertibleFunction`. Kept the `u_` prefix (load-bearing hint). (`2026/05/27/demo21-22-pipeline-dataclasses.md`)

- **Viz shaders de-duplicated by identity `(color source) × (animated project()?)`, not by the object drawn.** Shaders were named after the object (triangle/ground/cube) but their real identity is the color/animation combo; `project()` was byte-identical across most. Unifying fixed a real bug (perspective `axis.vert` used `max(1.0,…)` and never animated). GLSL 330 has no `#include`, so composition is a forward-declared `vec4 project(vec4)` + snippet source appended at compile time. (`2026/05/27/mvpviz-shared-shaders.md`)

- **`upload_mvp`/`_upload_frustum` renamed to `set_uniforms`.** "Upload" is wrong for `glUniform*` — the spec verb is "specify"; it updates cheap driver-side program state, not GPU memory. Reserve "upload" for `glBufferData`/`glTexImage2D`/PBOs, which actually move bytes. Also dropped the `mvp` qualifier since the functions set a varying uniform list. (`2026/05/27/notes-uniform-terminology.md`)

## Investigations that recommended NOT rewriting

Kept because the *conclusion* (don't touch it) is a decision someone will otherwise re-propose:

- **The matplotlib plot code should NOT adopt the GL side's `InvertibleFunction`/`Vector2` vocabulary.** Two measurements overturned the premise: the book **never shows this code** (0 `literalinclude`/`doc-region` refs — it's build tooling, only its SVG output is consumed), and the ugly nested-splat is ONE expression, not a pervasive style (50 of 51 call sites already compose via a list). Only added a small `compose()` at one site. Gate: 170 SVGs pixel-identical. (`2026/07/19/investigate-plotting-transform-model.md`)
